# MVP-дисциплина — ядро прежде обвязки

Стек: любой продуктовый прототип соло-разработчика.

## Контекст
Post-mortem умершего прототипа: почти все ~50 коммитов ушли в обвязку (соцфункции, обработка медиа, облачная инфраструктура, админ-страницы), а ядро ценности продукта так и осталось на мок-данных. Проект пришлось начинать заново.

## Уроки
1. **Сначала ядро, потом всё остальное.** Ни одной строчки второстепенных подсистем (соцслой, лента, сообщения), пока автор сам не пользуется ядром продукта в реальной жизни.
2. **Жёсткие этапы вместо «всё обязательно».** Следующий этап открывается только когда предыдущий работает в реальном использовании. Список «чего сознательно НЕТ до этапа N» фиксируется письменно — это забор от расползания.
3. **Никакой тяжёлой инфраструктуры на ноль пользователей.** Вместо документной БД + объектного хранилища + вебсокетов — SQLite + docker + файлы на диске + дешёвый VPS. Усложнять только когда реально упёрлись.
4. **Один критерий MVP:** автор сам ежедневно пользуется продуктом. Всё, что не приближает к этому, — в поздний этап.

## Как применять
При /start нового прототипа: зафиксировать этапы и список «чего НЕТ» в DECISIONS.md/CLAUDE.md как жёсткое правило, которое агент проверяет при планировании каждой задачи.
