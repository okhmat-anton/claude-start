# Команда /start — инициализация системы самосовершенствования

Ты — архитектор рабочего окружения Claude Code. Твоя задача: через короткое интервью развернуть (или аккуратно обновить) полную систему файлов, скиллов, хуков и агентов для продуктивной работы над проектом.

Работай СТРОГО по фазам. Не пропускай фазы и не начинай генерацию раньше времени.

---

## ФАЗА 0 — Определение режима

1. Проверь наличие в корне проекта: `CLAUDE.md`, директории `.claude/`, файлов `LESSONS.md`, `ACCEPTANCE.md`, `DECISIONS.md`.
2. Если НИЧЕГО из этого нет → режим **NEW** (новый проект).
3. Если хоть что-то есть → режим **UPGRADE** (существующий проект).
4. В режиме UPGRADE: прочитай все существующие файлы конфигурации и запомни их содержимое. Ты будешь ОБЯЗАН сохранить накопленные знания (уроки, решения, правила) — их потеря недопустима.
5. Сообщи пользователю одним предложением, какой режим определён и почему.

## ФАЗА 1 — Создание файла интервью

Создай в корне проекта файл `PROJECT_INTERVIEW.md` со следующей структурой. Каждый вопрос — заголовок, под ним строка `Ответ:` (пустая). Вопросы сформулируй так, чтобы на них можно было ответить 1–5 словами.

В режиме UPGRADE: предзаполни ответы всем, что смог достоверно извлечь из существующих файлов и кода проекта, и пометь такие ответы суффиксом `(автозаполнено — проверь)`. Пользователь исправит неверное.

### 25 вопросов

**О продукте**
1. Что это за продукт? (одна фраза)
2. Кто основной пользователь?
3. Стадия проекта? (прототип / MVP / продакшн)

**Технологии**
4. Основной язык программирования?
5. Фреймворк / основные библиотеки?
6. База данных?
7. Пакетный менеджер? (npm / pnpm / pip / poetry / cargo …)
8. Стиль API? (REST / GraphQL / gRPC / нет API)

**Команды (точные строки для терминала)**
9. Команда запуска проекта?
10. Команда запуска всех тестов?
11. Команда линтера / форматтера?
12. Команда сборки? (если нет — «нет»)

**Правила кода**
13. Стиль именования? (camelCase / snake_case / свои правила)
14. Что категорически ЗАПРЕЩЕНО делать в коде? (например: any в TS, сырые SQL-запросы)
15. Какие директории нельзя трогать никогда?
16. Обязательные проверки перед завершением любой задачи? (тесты / линт / сборка / типы)

**Git и процесс**
17. Формат коммитов? (conventional commits / свободный / свой)
18. Стратегия веток? (main-only / feature-ветки / gitflow)
19. Что входит в Definition of Done для задачи?

**Окружение**
20. Как хранятся секреты? (.env / vault / нет секретов)
21. Есть ли деплой, и как он делается? (кратко или «нет»)

**Приоритеты и агенты**
22. Что важнее при компромиссах: скорость разработки, производительность или надёжность? (расставь: 1, 2, 3)
23. Главные текущие боли / повторяющиеся проблемы в проекте?
24. Нужен ли отдельный агент-ревьюер кода с накоплением памяти? (да / нет)
25. Язык общения Claude с тобой? (русский / английский)

В конце файла добавь блок:

```
---
СТАТУС: НЕ ЗАПОЛНЕНО
(поменяй на «ГОТОВО», когда ответишь на все вопросы)
```

## ФАЗА 2 — Ожидание заполнения (блокирующая)

1. После создания файла НЕ ДЕЛАЙ НИЧЕГО дальше.
2. Задай пользователю вопрос через инструмент AskUserQuestion с вариантами:
   - **«Да, заполнил»** — переходи к Фазе 3.
   - **«Нет, ещё нет»** — вежливо ответь «Хорошо, жду. Напиши, когда заполнишь, или запусти /start снова» и ЗАВЕРШИ работу. Ничего не генерируй.

## ФАЗА 3 — Валидация ответов (строгая)

1. Перечитай `PROJECT_INTERVIEW.md` с диска (не из памяти — пользователь его менял).
2. Проверь КАЖДЫЙ из 25 вопросов. Ответ считается ОТСУТСТВУЮЩИМ, если:
   - строка `Ответ:` пустая;
   - ответ — заглушка вида «?», «...», «todo», «потом»;
   - остался непроверенный маркер `(автозаполнено — проверь)` без правки пользователем — такой ответ считается принятым, НО перечисли эти пункты пользователю для явного подтверждения.
3. Если СТАТУС в конце файла не «ГОТОВО» — считай файл незаполненным.
4. **Если есть хоть один отсутствующий ответ:**
   - выведи нумерованный список незаполненных вопросов;
   - НЕ генерируй никакие файлы, даже частично;
   - вернись к Фазе 2 (снова спроси «Да, заполнил / Нет») и повторяй цикл валидации, пока все 25 ответов не будут заполнены.
5. Только когда все 25 ответов есть — переходи к Фазе 4.

## ФАЗА 4 — Генерация системы

Сначала выведи пользователю короткий план: какие файлы будут созданы (режим NEW) или изменены/сохранены (режим UPGRADE). В режиме UPGRADE дождись подтверждения плана через AskUserQuestion («Применить план? Да / Нет»). Затем создай:

### 4.1. CLAUDE.md — только постоянные факты
Компактный (цель — до 60 строк). Включи ТОЛЬКО то, что нужно в каждой сессии:
- одно предложение о продукте (в. 1–3);
- стек и точные команды: запуск, тесты, линт, сборка (в. 4–12);
- жёсткие правила: запреты, неприкасаемые директории, стиль именования (в. 13–15);
- git-конвенции (в. 17–18);
- приоритеты компромиссов (в. 22);
- язык общения (в. 25);
- правило: «Каждый найденный баг → регрессионный тест»;
- правило: «После каждой значимой задачи запускай skill retro»;
- ссылки: «Процедуры — в .claude/skills/, уроки — в LESSONS.md, решения — в DECISIONS.md».

НЕ клади в CLAUDE.md процедуры, чеклисты и длинные инструкции — им место в skills.

### 4.2. Журналы
- `LESSONS.md` — с шапкой-инструкцией: формат записи урока (дата, контекст, урок, счётчик повторений) и правило продвижения: урок, повторившийся 2 раза, переносится в CLAUDE.md или превращается в skill, а из LESSONS.md удаляется.
- `DECISIONS.md` — журнал архитектурных решений (ADR-лайт): дата, решение, почему, отвергнутые альтернативы.
- `ACCEPTANCE.md` — чеклист готовности задачи, собранный из ответов на в. 16 и 19.

### 4.3. Skills (.claude/skills/)
- `retro/SKILL.md` — ретроспектива после задачи: что пошло не так → запись в LESSONS.md → проверка счётчиков повторений → продвижение зрелых уроков в CLAUDE.md/skills → чистка LESSONS.md.
- `done-check/SKILL.md` — прогон чеклиста из ACCEPTANCE.md перед завершением задачи.
- Если в ответе на в. 21 есть деплой — `release/SKILL.md` с процедурой релиза.
- У каждого skill: точное, конкретное поле description (когда именно его вызывать), тело до 100 строк.

### 4.4. Hooks (.claude/settings.json)
Собери из ответов на в. 10–11 и 15:
- `PostToolUse` (matcher: Edit|Write) → команда форматтера/линтера (в. 11);
- `Stop` → команда тестов (в. 10), если тесты существуют;
- `PreToolUse` (matcher: Bash) → блокировка `rm -rf` и записи в запрещённые директории (в. 15), скрипт положи в `.claude/hooks/`.
Если какой-то команды нет («нет») — соответствующий хук не создавай.

### 4.5. Subagents (.claude/agents/)
- Если в. 24 = «да»: `code-reviewer.md` с полем `memory: project` и инструкцией накапливать в памяти паттерны, конвенции и повторяющиеся ошибки проекта; доступ только на чтение.
- `explorer.md` — read-only агент для исследования кодовой базы и внешней документации, чтобы не засорять основной контекст.

### 4.6. Эмуляция агентской системы GitHub Copilot (VS Code)

Разверни поведение, повторяющее логику Copilot Agent Mode, средствами Claude Code:

**а) `agent-mode/SKILL.md`** — главный рабочий цикл (аналог Agent Mode). Description: «Используй для любой значимой задачи разработки». Тело skill:
1. **План-чеклист.** Перед любыми правками построй и покажи пользователю нумерованный todo-список шагов с чекбоксами `[ ]`. Крупную задачу дели на шаги ≤ 15 минут каждый.
2. **Пошаговое исполнение.** Выполняй шаги по порядку, после каждого отмечай `[x]` и показывай обновлённый чеклист одной строкой («Шаг 3/7 готов»). Если по ходу план меняется — покажи обновлённый план ПЕРЕД продолжением.
3. **Цикл самопроверки** (ключевое поведение Copilot): после каждого блока правок запускай команды проверки из CLAUDE.md (тесты → линт → сборка). Если есть ошибки — сам читай вывод, чини и запускай снова. Максимум 5 итераций автопочинки; если не сошлось — остановись, покажи оставшиеся ошибки и спроси пользователя, как действовать.
4. **Финальный отчёт.** В конце — компактная сводка: список изменённых файлов (по одному предложению на файл), что проверено, что осталось.

**б) Режимы чата** (аналог Ask / Edit / Agent):
- `ask-mode/SKILL.md` — «режим вопроса»: ТОЛЬКО отвечать и объяснять, ни одного изменения файлов, ни одной команды с побочными эффектами. Description: «когда пользователь просто спрашивает или пишет /ask».
- `edit-mode/SKILL.md` — «режим точечной правки»: менять ТОЛЬКО файлы, явно названные пользователем; не запускать команды, кроме проверочных; без расширения объёма задачи. Description: «когда пользователь просит поправить конкретный файл/место или пишет /edit».
- Полный агентский режим — это skill agent-mode из пункта (а).

**в) Правила с привязкой к путям** (аналог `.instructions.md` с `applyTo`): создай `.claude/rules/` и разложи туда правила из ответов на в. 13–15 и 23, которые касаются конкретных директорий или типов файлов, с glob-паттернами в frontmatter (например, правила для `src/legacy/**` или `**/*.test.ts`). Общие правила без привязки к путям остаются в CLAUDE.md.

**г) Зеркало для настоящего Copilot (опционально).** Спроси через AskUserQuestion: «Создать зеркальные файлы для GitHub Copilot, чтобы обе системы использовали одни инструкции? Да / Нет». Если «Да»:
- создай `.github/copilot-instructions.md` — краткая выжимка тех же фактов, что в CLAUDE.md;
- создай `.github/prompts/` с prompt-файлами, повторяющими ключевые skills (retro, done-check);
- добавь в skill `retro` пункт: «при изменении CLAUDE.md синхронизируй .github/copilot-instructions.md».

**д) В CLAUDE.md добавь правило:** «Любую значимую задачу разработки выполняй через skill agent-mode: план-чеклист → шаги → самопроверка → отчёт».

### 4.7. Правила режима UPGRADE (критично)
- НИЧЕГО не удаляй и не перезаписывай вслепую.
- Существующие LESSONS.md и DECISIONS.md: сохрани всё содержимое полностью, можно только добавить шапку-инструкцию, если её не было.
- Существующий CLAUDE.md: объедини — новые факты из интервью добавь, противоречия разреши в пользу свежих ответов пользователя, процедурные куски предложи вынести в skills (покажи, что именно переносишь).
- Существующие skills/hooks/agents: не трогай, только добавляй недостающие. При конфликте имён — спроси пользователя.
- Перед записью каждого изменяемого (не нового) файла покажи краткое описание изменений.

## ФАЗА 5 — Финал

1. Выведи дерево созданных/изменённых файлов с пометками [новый] / [обновлён] / [сохранён].
2. Удали `PROJECT_INTERVIEW.md`? — НЕТ, не удаляй автоматически. Спроси через AskUserQuestion: «Удалить файл интервью? Да / Оставить».
3. Заверши одним абзацем: как теперь работать (задача → done-check → retro) и напомни, что система будет умнеть через цикл LESSONS.md → CLAUDE.md/skills.

## Общие запреты

- Не генерируй файлы, пока валидация Фазы 3 не пройдена на 100%.
- Не выдумывай ответы за пользователя и не подставляй «разумные значения по умолчанию» вместо пустых ответов.
- Не раздувай CLAUDE.md: если сомневаешься, куда положить инструкцию — клади в skill.
- Все тексты генерируй на языке из ответа на вопрос 25.
